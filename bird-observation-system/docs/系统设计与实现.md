# 系统设计与实现

## 1. 系统建设目标

本课题围绕鸟类自动观测场景，设计并实现一个基于 YOLOv8 的鸟类检测与识别系统。系统以“可运行、可展示、可扩展”为核心目标，完成从数据采集、目标检测、珍稀预警、记录存储到统计分析与可视化展示的完整闭环。

与仅关注模型精度的单一算法实验不同，本系统强调算法模块与业务模块协同工作，既能够完成鸟类检测与类别识别任务，也能够满足毕业设计对系统完整性、工程实现性和展示效果的要求。

本系统主要实现以下目标：

- 实现基于 YOLOv8 的鸟类目标检测与识别
- 支持图片、视频、摄像头三种输入方式
- 实现珍稀鸟类分级预警
- 实现检测结果持久化存储
- 实现历史记录查询、统计分析和可视化展示
- 为后续模型优化与对比实验预留接口

## 2. 系统总体设计

### 2.1 设计思路

系统采用前后端分离架构。后端负责模型推理、业务逻辑处理、数据库读写和接口提供；前端负责页面展示、用户交互和统计图表渲染。该架构具有职责清晰、便于调试和适合答辩展示的优点。

整体流程如下：

1. 用户通过前端上传图片、视频或启动摄像头检测
2. 后端接收输入数据并调用 YOLOv8 模型进行推理
3. 系统对检测结果进行标准化处理
4. 根据鸟类信息表判断是否触发珍稀预警
5. 将检测记录和预警记录写入数据库
6. 统计服务从数据库中提取历史数据并生成图表所需结果
7. 前端通过接口获取统计结果并展示仪表盘、图表和记录信息

### 2.2 总体架构

系统总体可划分为五个层次：

- 模型层：负责 YOLOv8 权重加载、推理执行和结果标准化
- 服务层：负责图片检测、视频检测、摄像头检测、预警判断、记录存储和统计分析
- 接口层：负责对外提供 REST API
- 数据层：负责鸟类基础信息、检测记录、预警记录和系统日志的存储
- 展示层：负责首页仪表盘、记录列表和分析图表展示

## 3. 系统功能模块设计

### 3.1 数据集与训练模块

数据集采用 YOLO 检测格式组织，包含 `train`、`valid`、`test` 三个子集。系统在实验模块中提供训练、验证、测试、指标导出和对比实验脚本，用于支撑论文实验部分。

该模块实现内容包括：

- 数据集结构规范化
- 训练配置文件管理
- YOLOv8 基线训练脚本
- 验证与测试脚本
- 指标导出 CSV 脚本
- Precision、Recall、mAP、FPS 绘图脚本

### 3.2 实时检测与识别模块

实时检测与识别模块是系统的核心。该模块封装了 `BirdDetectionService`，实现对 YOLOv8 模型的统一调用。为了降低接口层与模型层的耦合，推理逻辑统一放在服务层中。

该模块主要支持以下三类输入：

- 静态图片检测
- 视频逐帧检测
- 摄像头实时检测

检测输出统一封装为标准 JSON 结构，包含类别编号、类别名称、置信度、边界框、是否珍稀、预警等级和结果保存路径等字段。

### 3.3 珍稀鸟类预警模块

为了使系统不仅具备检测能力，还具备实际业务价值，系统实现了珍稀鸟类预警模块。该模块基于 `bird_species` 表中的基础信息判断鸟类是否属于珍稀类别，并结合检测置信度与保护等级生成预警结果。

预警等级划分如下：

- `none`：普通鸟类或置信度不足
- `medium`：重点关注鸟类
- `high`：珍稀保护鸟类或持续出现的高风险情况

系统在检测后自动写入 `alert_records` 表，并提供最近预警查询接口，便于前端进行弹窗提醒和预警列表展示。

### 3.4 记录管理模块

记录管理模块负责将每次检测结果持久化存储到数据库中。其核心价值在于使系统从“实时识别工具”扩展为“可追溯的数据管理平台”。

该模块主要存储：

- 输入来源类型
- 原始文件名
- 识别到的鸟类类别
- 置信度与边界框
- 检测时间与位置信息
- 是否触发预警及预警等级
- 检测结果图像路径

通过记录管理模块，系统能够支持历史记录查询、详情查看和统计分析。

### 3.5 统计分析模块

统计分析模块面向系统展示和论文分析需求设计。其主要功能不是进行复杂生态建模，而是根据历史检测记录和预警数据生成可解释的统计结果。

当前已实现的统计包括：

- 总检测次数统计
- 今日检测次数统计
- 按鸟类类别统计频次
- 按日期统计趋势
- 珍稀鸟类预警统计
- 基础迁徙趋势模拟统计

统计结果以适合 ECharts 的 JSON 结构返回，便于前端直接绘制柱状图、折线图和饼图。

### 3.6 可视化展示模块

前端采用 Vue 3、Element Plus 和 ECharts 实现。首页仪表盘集中展示系统核心指标和图表信息，用于体现系统整体闭环与展示效果。

当前可视化内容包括：

- 总检测次数
- 今日检测次数
- 珍稀鸟类次数
- 预警次数
- 鸟类类别频次柱状图
- 检测趋势折线图
- 珍稀预警占比饼图
- 最近预警列表

## 4. 关键技术实现

### 4.1 YOLOv8 检测模型集成

YOLOv8 由 Ultralytics 提供，具备较好的检测精度与工程易用性。本系统选择 YOLOv8 作为核心检测模型，主要考虑以下因素：

- 支持目标检测任务，接口统一
- 推理调用简单，便于工程集成
- 模型训练、验证、测试工具链完整
- 适合本科毕业设计的开发周期与实验要求

在系统中，YOLOv8 模型加载、推理调用和结果可视化均由服务层统一封装，避免在接口层直接编写模型代码，从而提高系统的可维护性。

### 4.2 后端接口设计

后端采用 FastAPI 实现，具备接口清晰、开发效率高、自动生成 Swagger 文档等优点，适合本科毕业设计项目快速构建和演示。

系统接口主要包括三类：

- 检测接口：图片、视频、摄像头
- 管理接口：记录查询、预警查询
- 统计接口：概览统计、趋势分析、频次分析

通过统一响应结构，前后端字段命名保持一致，便于联调和后续扩展。

### 4.3 数据库设计

数据库采用 SQLAlchemy ORM 建模，开发阶段默认使用 SQLite，部署时兼容 MySQL。系统核心数据表包括：

- `bird_species`：鸟类基础信息表
- `detection_records`：检测记录表
- `alert_records`：预警记录表
- `system_logs`：系统日志表

该设计既能满足当前系统运行需要，也便于后续扩展更多字段，例如空间位置、迁徙区域或人工审核状态。

### 4.4 前端可视化实现

前端使用 Vue 3 进行页面构建，Element Plus 负责基础组件，ECharts 负责图表绘制。前端通过 Axios 请求后端接口，并结合路由组织页面结构。

这种实现方式具有以下优点：

- 组件化程度高，页面结构清晰
- 图表展示能力强，适合答辩演示
- 易于继续扩展图片检测页、视频检测页和历史记录页

## 5. 数据流程设计

系统的数据流程可概括为：

1. 输入数据进入后端
2. 检测服务完成目标检测与识别
3. 预警服务完成珍稀匹配和等级判断
4. 记录服务完成数据库持久化
5. 统计服务完成历史数据聚合
6. 前端获取数据后进行可视化展示

在这一流程中，模型推理、预警逻辑、数据记录和统计分析均已实现相互衔接，形成完整闭环。

## 6. 系统实现特点

本系统的实现具有以下特点：

### 6.1 强调完整性而非单点算法演示

系统不仅提供检测模型，还实现了预警、存储、统计、可视化等功能，使项目更符合毕业设计“系统设计与实现”的要求。

### 6.2 强调工程可运行性

系统采用单体式前后端分离方案，不引入微服务、消息队列和容器编排等复杂架构，部署难度低，适合本科毕业设计使用。

### 6.3 强调论文支撑能力

项目提供训练脚本、验证脚本、对比实验脚本、指标导出脚本和绘图脚本，便于直接服务于论文实验章节写作。

### 6.4 强调可扩展性

系统预留了模型替换、预警规则调整、数据库字段扩展和多模型对比实验接口，便于后续继续完善。

## 7. 存在的问题与改进方向

尽管系统已完成核心闭环，但仍存在进一步优化空间：

- 部分复杂背景下的小目标鸟类识别仍具有挑战
- 类别长尾问题会影响少样本类别检测效果
- 当前迁徙趋势分析仍以统计模拟为主，生态建模深度有限
- 当前训练环境主要依赖 CPU，完整训练耗时较长

后续可从以下方向继续改进：

- 引入注意力机制或小目标优化方法
- 使用类别重加权、重采样等方法改善长尾类别性能
- 引入更高质量、更细粒度的数据集
- 在具备 GPU 环境时进一步提升训练与推理效率
- 扩展多摄像头或多区域联合监测能力
